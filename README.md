# AI Engineering Projects

> 将模糊的业务需求落地为可运行的自动化方案。

本仓库包含四个独立的 AI 工程项目，覆盖客服系统的质量评估、异常检测、趋势分析与知识库治理。每个项目从原始数据出发，经过方案设计、代码实现、结果验证到报告输出，形成完整交付链路。

---

## 项目一览

### 1. 客服自动回复质量评估

#### 任务需求

团队上线了"客服自动回复"功能，需要评估回复质量。业务方要求比较模糊——"准确、有用、语气好、不能瞎编"，需要把这套模糊描述变成一套可运行的评估方案，包含：

- 定义至少 3 个可自动评估的指标，说明量化方式和选择理由
- 对 20 条自动回复逐条评分
- 评估在哪些 case 上可能不准及改进方向
- 输出评估报告（整体得分、各指标分布、最差 3 条 case 分析）

#### 方案设计

借鉴 RAGAS（Faithfulness、Answer Relevance）和 G-Eval Chain-of-Thought Rubric（Helpfulness、Tone & Empathy）两个学术框架，设计 **4 维度加权评分体系**：

| 维度 | 权重 | 量化方法 | 对应业务要求 |
|------|------|---------|-------------|
| Faithfulness（事实忠实性） | 25% | 拆解为原子陈述，逐条验证支持率 | 不瞎编、准确 |
| Answer Relevance（答案相关性） | 25% | 回复与问题的语义对齐度 | 准确 |
| Helpfulness（有用性/可操作性） | 30% | 分析回复是"帮用户做"还是"让用户自己去做" | 有用 |
| Tone & Empathy（语气与共情） | 20% | 识别用户情绪并评估共情程度 | 语气好 |

构建 **双模式评估流水线**：Mock 模式基于 human_ref 标注映射（无需 API Key，可离线运行），Real 模式调用 DeepSeek/OpenAI API 进行结构化评分。每条回复从 4 个维度打分（1-5 分），加权计算综合得分。

#### 结果展示

- **综合均分 4.04 / 5.0**（良好），最高分 4.65，最低分 3.07
- 最大短板：**Helpfulness（有用性）均分仅 3.7**，远低于其他维度
- 最差 3 条 case 共性：**"正确但没用"**——信息没错，但让用户自己操作，没有主动服务意识
  - case_03（退款到账 3.07 分）：只给通用退款时间表，没帮用户查实际进度
  - case_06（优惠券 3.38 分）：罗列可能原因但没帮用户排查
  - case_20（退货流程 3.50 分）：重复用户说搞不懂的流程，答非所问

### 2. 客服回复幻觉检测

#### 任务需求

团队智能客服系统偶尔会"说瞎话"——编造不存在的优惠政策、给出错误的退货地址、杜撰产品参数。业务方需要一个工具来批量检测这些"幻觉"回复，包含：

- 定义幻觉分类体系：哪些算幻觉？分几类？每类严重程度如何？
- 对 20 条客服回复逐条检测，标注结果
- 与人工标注结果（ground_truth）对比，计算漏检和误报
- 分析哪些 case 容易误判及原因

#### 方案设计

基于 RAG Faithfulness 评估框架和 Intrinsic/Extrinsic 幻觉分类法，设计 **6 类幻觉分类体系**（C1-C6），覆盖参数编造、政策编造、能力越界、信息编造、安全误导、信息遗漏，按严重程度分级（Critical/High/Medium/Low）。

构建 **Rule + LLM 混合检测架构**：

- **规则层**：3 条确定性规则——KB-Empty（知识库为空时检测能力越界）、Numeric-Conflict（数值参数矛盾检测）、Keyword-Negation（否定表述反推检测），置信度 0.70-0.95
- **LLM 层**：支持 DeepSeek API 和 Mock 模式，将回复拆解为原子陈述（claims），逐条与知识库对比验证
- **融合层**：Confidence-Based Weighted Fusion——规则置信度 ≥0.90 直接采纳，规则与 LLM 一致取 max，冲突时高置信度优先

输出逐条检测结果（含证据链、原子陈述级分析）+ 混淆矩阵 + 误判分析。

#### 结果展示

- **准确率 100%**（精确率 100% / 召回率 100% / F1 100%）
- 规则主导 5 条，LLM 主导 6 条，混合决策 9 条
- 识别出 5 类已知难点：
  - **h04 半对半错**：电子发票说对了，纸质发票编造了
  - **h07 能力越界+信息编造混合**：规则完全无法检出，完全依赖 LLM
  - **h09/h15 "未标注"≠"不存在"**：需语义推理
  - **h20 信息遗漏边界**：回复本身没错但遗漏关键信息，最易漏检

### 3. 客服工单趋势分析

#### 任务需求

客服主管每天要关注大量工单数据，需要快速了解：最近哪些问题变多了？哪些异常需要关注？目前靠人工翻看工单列表效率很低，需要自动从工单数据中发现趋势和异常，包含：

- 定义分析维度，说明每个维度对主管决策的价值
- 对 50 条工单数据进行分析，输出结构化趋势报告
- 识别值得关注的异常信号，说明判断依据
- 以合适方式呈现分析结果（图表 / Dashboard / 报告均可）

#### 方案设计

用 **Streamlit** 构建交互式 Dashboard，覆盖 **7 个分析维度**：

| 维度 | 核心指标 | 决策价值 |
|------|---------|---------|
| 总览 | KPI 卡片（工单量/满意度/处理时长/解决率） | 一眼掌握全局 |
| 时间趋势 | 日工单量折线 + 7 日滚动平均 | 发现突变拐点 |
| 分类分析 | 帕累托排行 + 分类×优先级热力图 | 定位 20% 问题类型 |
| 处理效率 | 各分类处理时长 + 箱线图 | 发现流程瓶颈 |
| 满意度分析 | 分布直方图 + 各渠道低分率 | 定位服务洼地 |
| 渠道分析 | 渠道占比 + 综合对比表 | 优化资源配置 |
| **异常检测（核心）** | 三级递进引擎 + 业务规则 | 自动预警 |

异常检测采用 **三级组合引擎**：Z-score 2σ（主检测器）→ 7 日滚动平均偏离 30%（趋势检测器）→ IQR 箱线图 1.5×（鲁棒校验器），综合判定≥2 级命中为确认异常。额外业务规则补充：满意度连续下降 ≥3 天预警、未解决率异常预警、分类突增预警。

#### 结果展示

- 支付问题（32%）+ 退款退货（26%）**占比 58%**，符合帕累托原则
- 整体满意度均值 **2.36/5**（偏低），处理时长与满意度呈**显著负相关**
- 高优先级工单占比 **62%**，最长处理时长 **120h**（极端长尾）
- 自动检测到：
  - 🔴 **2 次满意度连续下降异常**（6/4→6/6、6/9→6/11）
  - 🔴 **退款退货未解决率异常**（38.5%，远超整体 16%）

### 4. 知识库质量检测

#### 任务需求

智能客服知识库经过一年多积累有 500+ 条 FAQ，但从未系统维护，可能存在内容过时、条目间说法矛盾、部分问题未覆盖等问题。业务方希望做一次知识库"体检"，包含：

- 定义"问题条目"的分类体系及每类的影响
- 自动扫描知识库，标记有问题条目并说明具体问题
- 给出具体改进建议（修改、合并、删除、新增等）
- 输出可交给业务方看的治理报告

#### 方案设计

参考 **DAMA-DMBOK**（数据质量 8 维模型）、**ISO 8000**（数据质量过程模型）、**KCS**（知识中心服务方法论）等国际标准，设计 **8 类问题分类体系**，按 P0→P2 三级优先级排序。

构建 **双轨检测架构**：

| 检测层 | 方法 | 覆盖问题类型 |
|--------|------|-------------|
| **规则引擎** | 结构化对比（数值/布尔/枚举）+ 精确/模糊匹配 | 过时、空答案、重复、矛盾 |
| **LLM 语义检测** | 语义矛盾分析 + 专业度评估 + 建议生成 | 语义矛盾、表达不专业、覆盖缺失 |

治理建议按操作类型区分：**update**（修改）、**merge**（合并）、**create**（新增）、**improve**（优化），每条问题附带具体修改内容和关联条目。输出 JSON 结构化数据 + Markdown 可读报告。

#### 结果展示

- **健康率 62.5%**，综合质量得分（CQS）**3.8 / 5.0（B 级良好）**
- 发现 **15 条问题条目**，共 17 个问题点：
  - **P0（立即处理）3 条**：条目与业务规则矛盾 + 空答案
  - **P1（尽快处理）11 条**：内容过时 7 条、条目间矛盾 4 条
  - **P2（常规优化）1 条**：表达不专业
- **最大问题：时效性**（7 条过时条目，维度评分仅 3/5）
- **退货政策**分类健康率最低（44.4%），**发票**分类 0.0%（2 条全有问题）
- 建议操作：update 10 条、merge 1 组、create 3 条、improve 1 条
